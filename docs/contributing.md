# Contribution Guide

This document covers the full contribution workflow for the `ask-me` skill. For the short
version, see [CONTRIBUTING.md](../CONTRIBUTING.md) at the root.

---

## Before you start

Read [AGENTS.md](../AGENTS.md) for project context — especially the "What you should NOT change"
section. Then read [docs/skill-design.md](skill-design.md) to understand why the skill is
structured the way it is. Changing SKILL.md without understanding the design intent tends
to create subtle regressions that are hard to detect without running evals.

---

## Project setup

You only need [uv](https://docs.astral.sh/uv/) — no `npm install`, no `pip install`.
uv handles everything inline via the script metadata in `scripts/run_evals.py`.

```bash
# Clone
git clone git@github.com:cedricbou/ask-me.git
cd ask-me

# Verify the setup works (no API calls)
uv run scripts/run_evals.py --dry-run
```

Expected output: a list of 6 eval names, then "6 evals would run."

## Contribution types

### 1. Improving SKILL.md phrasing or structure

The skill body is written to explain *why* things matter, not just what to do. When editing:

- Keep instructions in the imperative form (`Use the question tool`, not `You should use`)
- Explain the reasoning behind constraints, not just the constraint itself
- Don't add MUST/NEVER in all-caps without a comment explaining the failure mode they prevent
- Keep the body under 500 lines — if you're approaching the limit, move reference material to `references/`
- After editing, run the full eval suite and check nothing regressed

The description field in the frontmatter is a **trigger**, not a summary. It tells the agent
*when* to activate, not *what* the skill does internally. Don't summarize the workflow there —
it causes the agent to follow the description instead of the skill body.

### 2. Adding an eval scenario

Evals live in `evals/evals.json`. Each scenario tests a specific path through the skill flow.
See [docs/evals-guide.md](evals-guide.md) for the full schema and assertion-writing guide.

Quick checklist:
- [ ] Add the scenario to `evals/evals.json` with a unique `id` and descriptive `name`
- [ ] Add simulated user responses to `SIMULATED_USER_RESPONSES` in `scripts/run_evals.py`
  (the key is the eval `name`, the value is a list of strings — one per round)
- [ ] Validate structure: `uv run scripts/run_evals.py --dry-run`
- [ ] Run the new eval: `uv run scripts/run_evals.py --eval <your-eval-name>`
- [ ] Run all evals to check for regressions: `uv run scripts/run_evals.py`

### 3. Improving the eval runner

`scripts/run_evals.py` is a standalone uv script. Its dependencies are declared in the
`# /// script` block at the top — no `pyproject.toml` needed.

When adding a new assertion type:
1. Add the assertion logic to `_check_behavioral` or `_check_qualitative` in `run_evals.py`
2. Add a test case in `evals/evals.json` that exercises the new assertion
3. Make sure the assertion returns a useful `evidence` string — it appears in the report

### 4. Documentation changes

- `README.md` — user-facing, keep it scannable
- `AGENTS.md` — agent-facing, keep it factual and structured
- `CONTRIBUTING.md` (root) — short version only, link to this file for details
- `docs/contributing.md` — this file
- `docs/evals-guide.md` — eval system documentation
- `docs/skill-design.md` — design decisions, should be updated when a design decision changes

`CHANGELOG.md` is auto-generated by semantic-release — do not edit it.

## Commit format

All commits must follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<optional scope>): <description>

[optional body]

[optional footer]
```

**Types that affect versioning:**

| Type | Version bump | When to use |
|---|---|---|
| `feat` | minor | New capability in the skill or tooling |
| `fix` | patch | Corrects incorrect behavior |
| `feat!` or `BREAKING CHANGE:` footer | major | Incompatible change to the skill interface |

**Types that don't bump version:**

| Type | When to use |
|---|---|
| `docs` | Documentation only |
| `test` | Adding or updating evals |
| `refactor` | Code restructure without behavior change |
| `chore` | Tooling, deps, CI config |

Examples:

```
feat: add gotchas section to Phase 1 questioning guidelines
fix: confirmation question now reliably offers all four paths
docs: add skill-design section on description field anti-patterns
test: add eval for multilingual confirmation step
refactor: extract tool call sequence helpers in run_evals.py
```

## Pull request workflow

1. Fork and create a feature branch: `git checkout -b feat/your-feature`
2. Make your changes
3. Run all evals: `uv run scripts/run_evals.py`
4. Commit with Conventional Commits
5. Push and open a PR against `main`
6. Describe what changed and why in the PR body

There are no automated CI checks yet beyond semantic-release on merge. We rely on evals
being run locally before opening a PR.

---

## Release process

Releases are fully automated via semantic-release:

1. Merge to `main`
2. semantic-release analyzes commit messages since last release
3. Determines the version bump (major/minor/patch) or skips if no releasable commits
4. Updates `CHANGELOG.md`
5. Creates a GitHub Release and tag

You don't need to do anything. Just merge and semantic-release handles the rest.

The release workflow requires a `GITHUB_TOKEN` secret — this is automatically available
in GitHub Actions on the `main` branch.
