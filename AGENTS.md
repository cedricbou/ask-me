# AGENTS.md — Context for AI Agents

This file is for AI agents working on this repository. Read it before making changes.

---

## What this project is

`ask-me` is a single-skill repository. The skill teaches an agent to frame a development
task through structured questioning (using the `question` tool) before writing any code —
all within a single conversation session.

The skill is published at `cedricbou/ask-me` and installable via:
```bash
npx skills add cedricbou/ask-me
```

## Core design intent

Read `docs/skill-design.md` for the full reasoning. Key points:

1. **`question` tool is mandatory for all user interaction.** The skill works because every
   interaction goes through the `question` tool — this keeps the session open (no new request
   needed) and ensures structured, efficient exchanges. Do not add instructions that bypass this.

2. **No rigid MUST/NEVER rules in SKILL.md.** The skill is written to explain *why* things
   matter and trust the agent's judgment. Heavy-handed rules make the skill brittle. See
   `docs/skill-design.md#why-no-must-rules` for the reasoning.

3. **The description field is a trigger, not a summary.** The YAML `description` in SKILL.md
   tells the agent WHEN to use the skill. It intentionally does not summarize the workflow — that
   would cause the agent to follow the description instead of reading the full skill body.

4. **Four phases, one session.** Questioning → Confirmation → (optional) Review → Implementation
   or Documentation. The session never ends in the middle; the `question` tool keeps it alive.

## How to work on this project

### Modifying the skill behavior

1. Read `docs/skill-design.md` before touching `SKILL.md`
2. Make your change
3. Run the evals: `uv run scripts/run_evals.py`
4. If behavior changes, update `evals/evals.json` to reflect the new expected behavior
5. Commit with a Conventional Commit message (feat/fix/docs/refactor/test)

### Adding a new eval scenario

1. Add the scenario to `evals/evals.json` (see `docs/evals-guide.md` for the schema)
2. Add simulated user responses to `SIMULATED_USER_RESPONSES` in `scripts/run_evals.py`
3. Run `uv run scripts/run_evals.py --dry-run` to validate structure
4. Run `uv run scripts/run_evals.py --eval <your-new-eval-name>` to test it

### Releases

Releases are automated. Push to `main` with Conventional Commits — semantic-release
handles versioning, CHANGELOG generation, and GitHub Release creation.

Do NOT manually edit `CHANGELOG.md`.

---

## What you should NOT change

- The `name` field in `SKILL.md` frontmatter — it's the install identifier
- The four-phase structure of the skill (Questioning → Confirmation → Review → Action)
- The requirement to use the `question` tool for all user interaction
- The `metadata.internal` flag (not set — this skill is public)

If you believe one of these should change, open an issue first.

## Running evals

```bash
# Auto-detects provider from env var (first match wins):
#   ANTHROPIC_API_KEY  → anthropic
#   OPENROUTER_API_KEY → openrouter
#   ZENCODE_API_KEY    → zencode
#   LLM_API_KEY        → custom (also set LLM_BASE_URL + LLM_MODEL)

export OPENROUTER_API_KEY=sk-or-...
uv run scripts/run_evals.py

# Force a provider
uv run scripts/run_evals.py --provider openrouter

# Override model
uv run scripts/run_evals.py --model anthropic/claude-opus-4-5

# Dry run (no API)
uv run scripts/run_evals.py --dry-run

# Write a full Markdown transcript for manual review
uv run scripts/run_evals.py --transcript transcripts/run.md
```

Exit code 0 = all pass. Exit code 1 = at least one failure.

The `--transcript` flag writes a human-readable Markdown file with the full conversation of
every simulated session — agent text, question tool calls with expanded options, and simulated
user replies. Use it to verify that the questions and plans the agent produces are actually
good, not just structurally correct. Transcript files are gitignored.
